# Бекап фото из Photos.app на Яндекс.Диск

Заливает на Яндекс.Диск только новые фото (jpg/jpeg/heic/png/dng) из папки
`originals` библиотеки Photos.app. Новизна определяется по SHA1-хешу файла,
список уже залитых хешей хранится в `manifest.json`.

## 1. Установка зависимостей

```bash
pip3 install requests python-dotenv Pillow pillow-heif
```

`Pillow` и `pillow-heif` нужны для чтения даты съёмки из EXIF (используется
для читаемых имён файлов — см. раздел 3).

## 2. Получение OAuth-токена Яндекс.Диска

1. Открой https://oauth.yandex.ru/client/new/
2. Название — любое, например `photo-backup`
3. В разделе "Платформы" выбери **Веб-сервисы**, Redirect URI укажи:
   `https://oauth.yandex.ru/verification_code`
4. В разделе "Яндекс.Диск REST API" поставь галочки:
   - "Запись в любом месте на Диске" (или уже-, если хочешь ограничить доступ только к создаваемой папке — но такого скоупа нет, доступен либо app_folder, либо весь диск)
5. Сохрани приложение, скопируй **ClientID**
6. Перейди по ссылке (подставь свой ClientID):
   ```
   https://oauth.yandex.ru/authorize?response_type=token&client_id=ТВОЙ_CLIENT_ID
   ```
7. Залогинься, разреши доступ — тебя перебросит на URL вида
   `https://oauth.yandex.ru/verification_code#access_token=AgAAAA...&token_type=bearer&expires_in=31536000`
8. Скопируй значение `access_token` — это и есть твой токен. Действует ~1 год (`expires_in` в секундах).

## 3. Настройка через .env

Скопируй шаблон и заполни своими значениями:

```bash
cp .env.example .env
```

Открой `.env` и впиши токен. `SOURCE_DIR` можно оставить пустым — скрипт сам
подставит `~/Pictures/Медиатека Фото.photoslibrary/originals` для текущего
пользователя macOS:

```
YADISK_TOKEN=AgAAAA...твой_токен
SOURCE_DIR=
DISK_BASE_PATH=/Backup/Photos
MANIFEST_PATH=manifest.json
PHOTO_EXTENSIONS=.jpg,.jpeg,.heic,.png,.dng
USE_EXIF_NAMES=true
UPLOAD_DELAY_SECONDS=0.5
```

Если медиатека лежит не в стандартном месте — пропиши свой путь явно, можно
с `~`:

```
SOURCE_DIR=~/Pictures/Медиатека Фото.photoslibrary/originals
```

`.env` уже добавлен в `.gitignore` — он не попадёт в git, если решишь оформить
проект в репозиторий.

### Читаемые имена файлов (USE_EXIF_NAMES)

По умолчанию включено (`USE_EXIF_NAMES=true`). Вместо UUID, которые Photos.app
даёт оригиналам, на Диске файлы получают имя по дате съёмки из EXIF:

```
2020-10-15_14-32-08.jpg
```

Год/месяц в пути на Диске тоже берутся из EXIF-даты, а не из даты модификации
файла на диске.

Если в файле нет EXIF-даты (бывает у скриншотов, у части конвертированных
файлов) — скрипт откатывается на оригинальное имя файла (UUID) и дату
модификации файла, чтобы ничего не падало и не терялось.

Если несколько кадров сняты в одну и ту же секунду (серийная съёмка) —
к имени добавляется суффикс: `2020-10-15_14-32-08_2.jpg`, `_3` и т.д. Эта
защита от коллизий работает в том числе между разными запусками скрипта
(сверяется с manifest.json), так что повторный запуск через 2-3 недели не
перепутает и не перезатрёт уже залитые файлы.

Если хочешь оставить оригинальные UUID-имена — поставь `USE_EXIF_NAMES=false`
в `.env`.

### Пауза между файлами и таймауты (UPLOAD_DELAY_SECONDS)

По умолчанию между заливкой файлов скрипт ждёт 0.5 секунды
(`UPLOAD_DELAY_SECONDS=0.5` в `.env`) — это защита от срабатывания rate limit
Яндекс.Диска при заливке тысяч файлов подряд. Можно увеличить, если всё равно
будут попадаться ошибки `429`, или поставить `0`, чтобы убрать паузу совсем.

Если раньше казалось, что скрипт "зависает" на каком-то файле — скорее всего,
дело было в сетевом запросе без таймаута (создание папки на Диске), который
мог ждать ответа бесконечно при малейшей сетевой заминке. Теперь у всех
сетевых запросов есть таймаут и retry с увеличивающейся паузой, так что
скрипт сам восстановится после кратковременных сетевых проблем вместо
зависания:
- перед каждой загрузкой файла печатается `заливаю...`, и после неё — сколько
  секунд это заняло
- таймаут на загрузку одного файла — 120 секунд, на остальные запросы к
  API — 30 секунд; при превышении скрипт сообщает об этом и пробует снова

## 4. Запуск

```bash
# Сначала проверь, что увидит скрипт, ничего не заливая:
python3 backup_photos.py --dry-run

# Реальная заливка:
python3 backup_photos.py
```

Скрипт сохраняет манифест после каждого успешно залитого файла — если прервать
скрипт (Ctrl+C, обрыв сети), уже залитые файлы не зальются повторно при
следующем запуске.

## 5. Восстановление manifest.json

Если файл потерян/удалён — можно пересобрать его, опросив сам Яндекс.Диск
(на основе md5, который Диск хранит в метаданных файлов):

```bash
python3 backup_photos.py --rebuild-manifest
```

После rebuild первый обычный запуск может посчитать уже залитые файлы
"новыми" (т.к. локально считается sha1, а с Диска восстановлен md5) — это не
страшно: Яндекс.Диск не даст создать дубликат с тем же именем
(`overwrite=false` вернёт 409), и файл просто корректно отметится в манифесте.

---

Скрипт не привязан к расписанию — запускается вручную, когда удобно (после
поездки, раз в 2-3 недели и т.п.). Манифест защищает от повторной заливки уже
закачанных файлов, так что можно запускать сколько угодно раз — лишнего не
зальётся.

## Структура файлов на Диске

С включёнными читаемыми именами (`USE_EXIF_NAMES=true`, по умолчанию):

```
/Backup/Photos/
  2018/
    11/
      2018-11-03_09-15-22.jpg
  2020/
    10/
      2020-10-15_14-32-08.jpg
      2020-10-15_14-32-08_2.jpg   <- коллизия: второй кадр в ту же секунду
```

Год/месяц и имя файла берутся из EXIF-даты съёмки. Если у файла нет
EXIF-даты — он попадает в папку по дате модификации файла, а имя остаётся
оригинальным (UUID, как его назвал Photos.app):

```
/Backup/Photos/
  2026/
    06/
      6482611C-9991-456E-9971-F887587D694A.jpeg
```

Подробнее — см. раздел про `USE_EXIF_NAMES` выше.
